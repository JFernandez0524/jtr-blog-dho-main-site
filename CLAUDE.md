## What this project is

Professional real estate authority site for Jose Fernandez (RE/MAX NJ), targeting inherited property, foreclosure, and as-is home sale leads. Next.js 15 App Router frontend backed by AWS Amplify Gen 2 (DynamoDB + Lambda + Cognito).

## Commands

```bash
npm run dev       # Start dev server at localhost:3000
npm run build     # Production build
npm run lint      # ESLint via next lint

# Amplify backend (requires AWS credentials)
npx ampx sandbox --identifier jtr    # Deploy sandbox backend, generates amplify_outputs.json
npx ampx sandbox delete --identifier jtr  # Tear down sandbox
```

No test suite is configured. TypeScript type-checking: `npx tsc --noEmit`.

## Architecture

### Data flow for lead capture

Contact/valuation form (client) → `POST /api/contact` or `/api/valuation` (Next.js route) → rate limit + reCAPTCHA + Zod validation → write `ContactSubmission` to DynamoDB via `cookiesClient` → call `syncToGHL()` from `lib/ghl.ts` directly (HTTP to GHL API with 3-attempt exponential backoff) → update `ghlSyncStatus` on the DynamoDB record.

Admin panel at `/admin/leads` reads DynamoDB directly; protected by Cognito (Google OAuth) via `middleware.ts`. Failed syncs can be retried via `POST /api/admin/retry-sync`.

Note: `amplify/functions/ghl-contact/handler.ts` is a Lambda with equivalent GHL sync logic — it exists as a separately deployable path but is **not** invoked by the current API routes, which call `lib/ghl.ts` directly.

### AWS Amplify Gen 2 backend

- `amplify/data/resource.ts` — DynamoDB schema: `ContactSubmission`, `Lead`, `Project`, `RateLimit`
- `amplify/auth/resource.ts` — Cognito User Pool with Google OAuth
- `amplify/functions/ghl-contact/handler.ts` — Lambda with GHL sync logic (see note above)
- `amplify_outputs.json` — auto-generated by `ampx sandbox`; **do not edit manually**

`ContactSubmission.authorization`: `publicApiKey` for create/update (unauthenticated form submissions), `authenticated` for read/update/delete (admin).

### Amplify server-side utilities

`utils/amplify-utils.ts` exports three things used throughout server components and API routes:
- `cookiesClient` — typed DynamoDB client for server components/routes (uses cookies for auth)
- `runWithAmplifyServerContext` — wraps Amplify auth operations in server context (used in `middleware.ts` and `retry-sync`)
- `AuthGetCurrentUserServer()` — helper to get the current Cognito user server-side

### Content system

Two content types in `content/`:

**Blog posts** (`content/blog/*.mdx`) — available at `/blog/[slug]`. Frontmatter: `title`, `description`, `date`, `author`, `tags[]`, `image`.

**Location pages** (`content/locations/*.mdx`) — available at `/[locationSlug]` (e.g. `/inherited-property-freehold-nj`). Frontmatter: `town`, `county`, `title`, `metaDescription`, `heroHeading`, `heroSubtext`, `youtubeId` (optional). Slugs are the filename without `.mdx`.

Both are read at build time via `lib/mdx.ts` — key exports: `getAllPosts`, `getPostBySlug`, `getPostSlugs`, `getAllLocations`, `getLocationBySlug`, `getLocationSlugs`.

### Centralized configuration

`lib/config.ts` exports `siteConfig` — the single source of truth for contact info, nav links, service types, social URLs, hero copy. Never hardcode these values in components.

### GHL custom field IDs

`lib/ghl.ts` maps form data to GHL custom fields using hardcoded field IDs (e.g. `"ZvxM7eN2rOdgdV9OGKY6"` for message). These IDs correspond to specific fields in the GHL location config — do not change them without updating the GHL account.

### Next.js 15 async params pattern

All dynamic routes must `await params` — this is required by Next.js 15:

```typescript
export default async function Page({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;
}
```

### Client vs Server Components

Default: Server Components. Use `"use client"` only for interactivity (forms, state, browser APIs). Client Components: `ContactForm`, `PropertyValuationForm`, `Login`, `Logout`, `SafeRecaptchaProvider`, `ConfigureAmplifyClientSide`. The `SafeRecaptchaProvider` wrapper exists because `GoogleReCaptchaProvider` crashes in SSR context — always wrap reCAPTCHA usage in it.

### Security headers

`next.config.js` sets a strict CSP. If adding new external scripts/fonts/frames, update the CSP there — do not add `unsafe-eval`.

### Rate limiting

`lib/rateLimit.ts` uses the `RateLimit` DynamoDB model. API routes that accept public input should apply it.

### Pillar pages vs location pages

`/inherited-property-new-jersey`, `/foreclosure`, `/sell-as-is` are static pages in `app/` and take routing precedence. The MDX-driven location pages (`/inherited-property-{town}-nj`) live under `app/[locationSlug]/` — a dynamic catch-all that matches any path not claimed by a more specific `app/` route. Do not add new top-level routes that could collide with existing location slugs.
